History

React introduced JSX. It‚Äôs been battle-tested and preferred by many development teams regardless of the initial criticism. JSX is ‚Äújust javascript‚Äù and with this there‚Äôs high focus on component-oriented development and composition. After all, a React component is ‚Äújust a function‚Äù. The JSX style fits well in teams with deeper JavaScript knowledge, on top of jQuery, and the ES6+ features.

Forget about JSX when working with Vue.

When it comes to Vue, web developers with experience in PHP and jQuery appreciate Vue syntax, because Vue templates are more about HTML rather than JavaScript. So, yeah, when you work with developers who are used to make dynamic HTML templates with inline variables, control statements and loops, Vue is more appealing and understandable.

An interesting fact from history is that Vue‚Äôs default syntax is inspired by Angular 1, not PHP templating. Angular 1 just entered EOL (End of Life phase) but it played a key role in making the MEAN stack a viable alternative to the LAMP stack a decade ago.

Google Creative Labs was where Evan first started his work on Vue. As the project grew, the team started to use Angular 1. it had too many features that they didn‚Äôt need. Evan also didn‚Äôt like some of the design decisions that Angular 1 had. So, Evan started to work on a templating library just for his personal use. After six months, in February 2014, he officially released it as Vue.js. (source)

Knowing a bit of history is nice, but what are the key concepts to grasp in order smoothen the learning curve when it comes to Vue templates?

Another unique feature of Vue which resembles (more or less) children prop is in React facilitating components‚Äô inheritance and composition. In the PHP world, this would translate (more or less) to extends and blocks in twig.

Slots are a must-read and a must-understand feature which provides code flexibility and reusability.

üîó Refs [documentation]

These are similar to other refs, yes. They should be used carefully because it‚Äôs easy to introduce fishy code through lack of understanding of boundaries between the DOM and Vue.

Refs in Vue return an object of DOM elements and component instances. This means that there is a combination of different APIs. There are no limitations with regards to breaking the boundaries I just mentioned, so it could happend that Vue methods are invoked (instead of DOM such) causing unintentional and hard to debug changes in data.

Refs should be used for focus management, not state management.

The last important piece of advice I‚Äôd share for this topic is about how to safely access Vue refs without getting undefined.

State management

The concept of state is not as prominent in Vue as it is in React. For a refresher, React provides setState() function which does exactly what it is named after. Each change in state triggers re-rendering.

Vue provides data() function which returns reactive variables. Reactivity was the hardest to wrap my head around. Nested objects remain reactive and keep track of changes when passed around with default assignment operations. This is the best and worst feature of Vue in my personal opinion.

Therefore, my first recommendation to anyone experiencing issues with reactivity in nested objects: justjavascript.com. It‚Äôs a great resource for deeply understanding objects in JavaScript. In increased complexity of nested components and data structures in Vue, the knowledge on how to ‚Äúbreak reference‚Äù to avoid unintentional state changes is vital.

And when discussing complexity, here‚Äôs my second recommendation: consider carefully state management in Vue and Vuex. It‚Äôs not adding extra complexity and it‚Äôs also inspired by Flux similarly to Redux.

Vuex is more accessible than Redux and I‚Äôd definitely start with it in any new greenfield project. It‚Äôs way easier to scale up a Vue project with clearly managed state following battle-tested patterns.

Recently, I came across Pinia, but I haven‚Äôt tried or used it. And by the way, Vue 3 uses proxies which is a different approach to reactivity compared to Vue 2.

Managing reactive state is the most important skill to acquire.

Computed properties

Computed properties are a unique feature of Vue. They facilitate caching of reactive variables. I‚Äôve deliberately kept them for the end, because they are an optimisation technique which is easy to grasp when data() and methods concepts are mastered properly.
